3. CROSS-REGISTRY CONSISTENCY
--------------------------------------------------------------------------------

  Name overlap between IndicatorRegistry (24 names) and SMCRegistry
  (32 names): NONE. This is important -- app.strategy_builder.resolution
  resolves an SDL IndicatorSpec.type against IndicatorRegistry first,
  then SMCRegistry; a name present in both would be flagged "ambiguous"
  by StrategyValidator. With the current 24+32 real components, this
  can never happen (verified empty-set intersection). The "ambiguous"
  code path was still exercise-tested using synthetic fake registries
  in tests/strategy_builder/test_resolution.py, confirming the guard
  works even though it's currently unreachable with real data.

--------------------------------------------------------------------------------
4. STRATEGY BUILDER'S OWN REGISTRY (app.strategy_builder.StrategyRegistry)
--------------------------------------------------------------------------------

  Distinct from app.sdl.StrategyRegistry (filesystem-backed SDL document
  library) -- this registry holds built, in-memory StrategyModel objects,
  keyed by the source SDL strategy id. Same register/load/search/enable/
  disable/list shape as IndicatorRegistry/SMCRegistry.

  Verified this audit:
    register(model) -> load(id) round-trips to an identical StrategyModel
    register() without overwrite on a duplicate id -> raises
      StrategyRegistrationError
    disable(id) -> require_enabled(id) raises StrategyDisabledError
    enable(id) -> require_enabled(id) succeeds
